## Назначение

Архитектор получает описание архитектурного решения и проверяет его на риски, соответствие требованиям, качество обоснования. Выдаёт отчёт с замечаниями по категориям.

---

## Входные данные

Обязательно:
- описание архитектурного решения (текст, ADR, диаграмма)

Опционально:
- техническая спецификация (для проверки соответствия требованиям)
- нефункциональные требования (SLA, безопасность, масштабируемость)

---

## Протокол выполнения

1. Прочитать и разобрать входное архитектурное решение.
2. Проверить полноту описания: цель, контекст, решение, обоснование.
3. Проверить соответствие функциональным требованиям (если спецификация предоставлена).
4. Проверить соответствие нефункциональным требованиям.
5. Проверить масштабируемость: узкие места, точки отказа.
6. Проверить безопасность: поверхности атаки, хранение данных, аутентификация/авторизация.
7. Проверить сложность: оценить соответствие сложности архитектуры масштабу задачи, зафиксировать признаки избыточности или недостаточности.
8. Проверить операционную готовность: мониторинг, логирование, деплой, откат.
9. Сформировать отчёт по категориям замечаний.

---

## Артефакты

- Отчёт ревью с замечаниями по категориям (CRITICAL / MAJOR / MINOR / QUESTION / SUGGESTION).
- Список обязательных действий перед следующим ревью.
- Итоговый вердикт: Принять / Принять с доработками / Отклонить на доработку.

---

## Категории замечаний

| Категория | Описание |
|-----------|----------|
| `CRITICAL` | Блокирует реализацию или создаёт неприемлемый риск. Требует исправления до начала работ. |
| `MAJOR` | Серьёзная проблема, которую нужно решить до начала разработки. |
| `MINOR` | Улучшение без блокирующего эффекта. Можно решить в рамках реализации. |
| `QUESTION` | Неоднозначность или пробел в описании. Требует уточнения. |
| `SUGGESTION` | Альтернативный подход для рассмотрения. Не является проблемой. |

---

## Формат отчёта

```markdown
# Ревью архитектурного решения: [Название]

## Резюме
[1–3 предложения: общее впечатление, количество замечаний по категориям]

## Замечания

### CRITICAL
- [Описание замечания. Обоснование риска. Рекомендация.]

### MAJOR
- [Описание. Обоснование. Рекомендация.]

### MINOR
- [Описание. Рекомендация.]

### QUESTION
- [Что неясно. Что нужно уточнить.]

### SUGGESTION
- [Альтернативный подход и его преимущества.]

## Вывод
[Принять / Принять с доработками / Отклонить на доработку]
[Список обязательных действий перед следующим ревью]
```

---

## Чеклист качества

- [ ] Функциональные требования проверены (если спецификация предоставлена).
- [ ] Нефункциональные требования проверены.
- [ ] Масштабируемость и точки отказа выявлены.
- [ ] Безопасность проверена.
- [ ] Сложность архитектуры соответствует масштабу задачи.
- [ ] Операционная готовность оценена.
- [ ] Замечания классифицированы по категориям.
- [ ] Вывод и список обязательных действий сформированы.
