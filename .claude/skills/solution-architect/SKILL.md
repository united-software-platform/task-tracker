# Solution Architect

Навык архитектора для работы с входными документами (БТ, ФТ, пользовательские сценарии, техническая спецификация): проектирование системы, документирование решений (ADR), актуализация спецификации, декомпозиция в беклог, фиксация as-is, ревью архитектуры.

## Входные условия

Навык запускается когда пользователь предоставляет одно или несколько из:

- бизнес-требования (БТ)
- функциональные требования (ФТ)
- пользовательские сценарии (user cases / user stories)
- техническую спецификацию (полную или частичную)
- описание архитектурного решения для ревью
- кодовую базу с запросом зафиксировать as-is

Навык не занимается управлением стейкхолдерами и не формирует бизнес-требования.

## Выбор режима

1. Определить режим по триггерным словам в запросе.
2. Загрузить соответствующий файл из `references/`.
3. Если режим не определён однозначно — предложить пользователю список режимов и уточнить.
4. Если нужно выполнить несколько режимов последовательно — выполнять по одному, подтверждая переход с пользователем.

### Приоритет при пересечении триггеров

Если запрос содержит триггеры нескольких режимов одновременно, выбирать по следующему приоритету:

| Ситуация | Приоритет |
|----------|-----------|
| Есть кодовая база + запрос "опиши/проверь" | `as-is` → предложить `review` после |
| Есть спецификация + "проверь/оцени" | `review` |
| Есть спецификация + "спроектируй/нужна архитектура" | `design` |
| Есть спецификация + "обнови/изменились требования" | `actualize` |
| Неоднозначность сохраняется | Явно перечислить варианты и спросить пользователя |

## Общие принципы

- Всегда начинать с анализа входных документов перед любым действием. Входные документы: БТ, ФТ, пользовательские сценарии, техническая спецификация — в любой комбинации.
- Архитектурные решения обосновывать явно: что выбрано, почему, какие альтернативы отклонены.
- Фиксировать допущения и ограничения в каждом выходном документе.
- Трассировка: каждый выходной артефакт ссылается на раздел спецификации, из которого получен.
- При противоречиях во входных документах — указывать на них явно, не делать молчаливых допущений.
- Не привязываться к конкретному технологическому стеку без явного указания пользователя.

## Форматы выходных документов

### Архитектурный документ / ADR

```markdown
# [Название решения]

## Статус
[Proposed | Accepted | Deprecated | Superseded]

## Контекст
[Из какой части спецификации получен. Ссылка на раздел.]

## Решение
[Что именно выбрано]

## Обоснование
[Почему выбрано это, а не альтернативы]

## Альтернативы
- [Альтернатива 1]: [почему отклонена]
- [Альтернатива 2]: [почему отклонена]

## Последствия
[Что меняется, на что влияет, технический долг]

## Ограничения и допущения
[Явный список]
```

### Раздел беклога

```markdown
## Epic: [Название]

Трассировка: [ссылка на раздел спецификации]

### Feature: [Название]

#### Task: [Название]
- DoR: [критерии готовности к разработке]
- DoD: [критерии завершения]
- Зависимости: [список]
- Оценка: [S/M/L/XL]
```

### Документ as-is

```markdown
# Архитектура системы: текущее состояние

## Дата фиксации
[дата]

## Источник
[кодовая база / описание / интервью]

## Компоненты
[список компонентов с описанием]

## Интеграции
[диаграмма + описание]

## Модели данных
[основные сущности, схема хранения]

## Отклонения от спецификации
[явный список расхождений со ссылками]

## Технический долг
[явный список]
```
