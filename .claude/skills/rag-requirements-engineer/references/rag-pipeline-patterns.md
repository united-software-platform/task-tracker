# RAG Pipeline Patterns — Технический справочник

Справочник по стратегиям, моделям и метрикам для проектирования RAG-пайплайнов. Использовать при составлении ТЗ — выбирать паттерн, обосновывать выбор, указывать trade-offs.

---

## 1. Стратегии чанкинга

Чанкинг — разбивка документов на фрагменты перед векторизацией. Выбор стратегии напрямую влияет на качество retrieval.

### 1.1 Fixed-size Chunking

**Описание:** Документ разбивается на фрагменты фиксированного размера (в токенах или символах) с опциональным перекрытием (overlap).

**Параметры:**
- `chunk_size`: 256–1024 токена (типично 512)
- `overlap`: 10–20% от chunk_size (типично 64–128 токенов)

**Когда применять:**
- Документы однородные по структуре (статьи, отчёты)
- Быстрый старт, нет данных о структуре документов
- Базовый baseline для последующего улучшения

**Trade-offs:**
| Плюсы | Минусы |
|-------|--------|
| Простота реализации | Разрывает предложения и абзацы |
| Предсказуемый размер чанков | Контекст может теряться на границах |
| Легко параметризовать | Не учитывает структуру документа |

**Рекомендуемые параметры для технических документов:** `chunk_size=512, overlap=64`

---

### 1.2 Semantic Chunking

**Описание:** Границы чанков определяются по семантическим разрывам — изменениям смысла между соседними предложениями, измеряемым косинусным расстоянием embeddings.

**Параметры:**
- `breakpoint_threshold`: 0.3–0.7 (порог косинусного расстояния для разбивки)
- `min_chunk_size`: минимальный размер чанка в токенах

**Когда применять:**
- Документы с нелинейной структурой, разными темами внутри одного файла
- Высокие требования к качеству retrieval
- Есть ресурсы на более долгую индексацию

**Trade-offs:**
| Плюсы | Минусы |
|-------|--------|
| Сохраняет семантическую целостность | Медленнее fixed-size |
| Лучший recall на тематических запросах | Непредсказуемый размер чанков |
| Меньше "разорванного" контекста | Требует embedding-модели на этапе ingestion |

---

### 1.3 Hierarchical Chunking (структурный)

**Описание:** Документ разбивается с учётом его структуры: заголовки H1/H2/H3 образуют иерархию, чанки соответствуют разделам.

**Параметры:**
- `header_levels`: какие уровни заголовков являются границами чанков
- `max_chunk_size`: максимальный размер раздела

**Когда применять:**
- Структурированные документы: Markdown, Word с заголовками, HTML
- Документация, нормативные акты, руководства
- Нужна навигация по структуре документа

**Trade-offs:**
| Плюсы | Минусы |
|-------|--------|
| Чанки соответствуют логическим единицам | Требует разметки структуры документа |
| Легко добавлять breadcrumb-контекст | Неравномерный размер чанков |
| Хорошо для FAQ и документации | Не работает для неструктурированного текста |

---

### 1.4 Parent-Child Chunking

**Описание:** Документ индексируется в двух гранулярностях: крупные "родительские" чанки хранятся для контекста, мелкие "дочерние" чанки используются для поиска. После retrieval дочерних чанков — в промпт передаётся родительский.

**Параметры:**
- `parent_chunk_size`: 1024–2048 токенов
- `child_chunk_size`: 128–256 токенов
- `overlap` для дочерних чанков: 20–30 токенов

**Когда применять:**
- Нужна высокая точность поиска (мелкие чанки) и богатый контекст (крупные)
- Документы с плотной информацией, где каждое предложение важно
- Есть возможность хранить два набора индексов

**Trade-offs:**
| Плюсы | Минусы |
|-------|--------|
| Лучший баланс precision/recall | Сложнее реализация |
| Богатый контекст в промпте | Больше памяти и хранилища |
| Мелкие чанки — точный поиск | Двойная индексация |

**Реализация в LlamaIndex:** `HierarchicalNodeParser` + `AutoMergingRetriever`

|--------|-------|-------------|---------------|------------|----------|
| intfloat/multilingual-e5-large | 100+ | 1024 | 560 MB | 64.6 (multilingual) | MIT |
| intfloat/multilingual-e5-base | 100+ | 768 | 278 MB | 62.5 | MIT |
| OpenAI text-embedding-3-large | 100+ | 3072 (настраиваем) | API | 64.6 | Проприетарная |

### 2.2 Рекомендации по выбору

**Только русский язык, локальное развёртывание:**
Выбрать `intfloat/multilingual-e5-large`. Отличное качество на русском, поддерживается sentence-transformers, MIT-лицензия. Требует prefix "query: " для запросов и "passage: " для документов.

**Русский + английский, локальное развёртывание, ограниченные ресурсы:**
Выбрать `intfloat/multilingual-e5-base`. Размерность 768, меньше памяти, качество ниже на ~2%.

**Максимальное качество, облако допустимо:**
Выбрать `OpenAI text-embedding-3-large`. Настраиваемая размерность, SOTA качество, но требует API-ключ и интернет.

**Русский язык, нужен hybrid search (dense + sparse):**
Выбрать `BAAI/bge-m3`. Единственная модель, генерирующая одновременно dense, sparse и ColBERT векторы.

### 2.3 Важные особенности моделей

**multilingual-e5 (large/base):**
- ОБЯЗАТЕЛЬНО использовать prefix: `"query: "` для запросов, `"passage: "` для документов
- Без prefix качество падает на 5–10%
- Нормализация embeddings обязательна для cosine similarity

**BGE-M3:**
- Поддерживает multi-vector retrieval (ColBERT), sparse retrieval и dense retrieval одновременно
- Оптимален для Qdrant с включённым sparse vectors

**all-MiniLM-L6-v2:**
- Только английский — не использовать для русскоязычных документов
- Подходит как baseline для быстрого прототипа на английских данных

## 3. Vector Databases

### 3.1 Сравнительная таблица

| База | Фильтрация | Hybrid Search | Масштабирование | Режим развёртывания | Лицензия |
|------|-----------|---------------|-----------------|--------------------|----|
| Qdrant | Payload фильтрация | Dense + Sparse (встроен) | Distributed | Docker, облако | Apache 2.0 |
| FAISS | Нет | Нет (внешний BM25) | Одна нода | In-process | MIT |
| Weaviate | Развитая | BM25 + Vector (встроен) | Distributed | Docker, облако | BSD-3 |
| Chroma | Базовая | Нет | Одна нода | In-process, сервер | Apache 2.0 |
| Milvus | Развитая | Dense + Sparse | Distributed | Docker, облако | Apache 2.0 |
| pgvector | SQL WHERE | Нет | PostgreSQL | Расширение PG | PostgreSQL |

### 3.2 Рекомендации по выбору

**Qdrant** — выбрать, если:
- Нужна payload-фильтрация (по метаданным документов)
- Планируется hybrid search (dense + sparse)
- Docker-развёртывание, нет Kubernetes
- Важна простота настройки и хорошая Python-библиотека

**FAISS** — выбрать, если:
- In-memory индекс, нет требований к персистентности
- Нет потребности в фильтрации
- Максимальная скорость поиска, минимальный overhead
- Не нужен отдельный сервис

**Weaviate** — выбрать, если:
- Нужны GraphQL-запросы и семантический поиск по схеме
- Развитые возможности фильтрации с разными типами данных
- Нужен встроенный модуль генерации (Generative module)

**pgvector** — выбрать, если:
- Уже используется PostgreSQL и нет желания добавлять новый сервис
- Датасет небольшой (до 1M векторов)
- Нужны сложные SQL-запросы с JOIN к реляционным данным

### 3.3 Qdrant: ключевые параметры конфигурации

```yaml
# Создание коллекции
collection_name: documents
vectors:
  size: 1024           # Размерность embedding-модели
  distance: Cosine     # Cosine, Dot, Euclid — выбирать по модели
  on_disk: false       # true для больших индексов (> RAM)

# HNSW параметры (влияют на качество/скорость поиска)
hnsw_config:
  m: 16               # Число рёбер в графе. Больше = точнее, медленнее
  ef_construct: 100   # Качество при построении. Больше = точнее, дольше
  full_scan_threshold: 10000  # До N точек — полный скан вместо HNSW

# Параметры поиска
search_params:
  ef: 128             # Точность поиска. >= top_k * 2 рекомендуется
  exact: false        # true = точный поиск (медленно, только для тестов)
```

## 4. Retrieval-стратегии

### 4.1 Dense Retrieval (векторный поиск)

**Описание:** Запрос кодируется embedding-моделью, ищутся ближайшие соседи в vector store.

**Когда применять:**
- Семантически похожие запросы ("расскажи про аутентификацию" → документ об auth)
- Мультиязычные запросы
- Запросы без точных ключевых слов

**Параметры:**
- `top_k`: 5–20 (зависит от reranker и длины контекста)
- `score_threshold`: минимальный score для включения в результат (опционально)

**Ограничения:** Плохо находит документы по точным терминам (коды ошибок, имена функций, ID).

---

### 4.2 Sparse Retrieval (BM25)

**Описание:** Классический TF-IDF / BM25 поиск по ключевым словам.

**Когда применять:**
- Запросы с точными терминами: "ошибка ERR_CODE_42", "функция process_document"
- Юридические и нормативные документы с точными формулировками
- Дополнение к dense для hybrid подхода

**Ограничения:** Не понимает семантику — "автомобиль" и "машина" — разные слова для BM25.

---

### 4.3 Hybrid Search (RRF Fusion)

**Описание:** Результаты dense и sparse retrieval объединяются через Reciprocal Rank Fusion (RRF).

**Формула RRF:**
```
RRF_score(doc) = Σ 1 / (k + rank(doc, ranker_i))
где k = 60 (константа, снижает влияние топ-1)
```

**Параметры:**
- `alpha`: вес dense vs sparse (0.0 = только sparse, 1.0 = только dense; 0.5–0.7 типично)
- `k`: константа RRF (обычно 60)
- `top_k` для каждого ranker: в 2x больше финального top_k

**Когда применять:**
- Документы смешанного типа (технические + описательные)
- Неизвестен характер запросов заранее
- Нужна устойчивость к разным типам запросов

**Реализация в Qdrant:**
```python
# Qdrant поддерживает hybrid search нативно через sparse vectors
# Требует BAAI/bge-m3 или отдельного BM25-индексатора (fastembed)
```

### 4.4 Query Rewriting и Expansion

**Описание:** Перед retrieval запрос пользователя переписывается или расширяется для улучшения покрытия.

**Техники:**
- **HyDE (Hypothetical Document Embedding):** LLM генерирует гипотетический документ-ответ, его embedding используется для поиска
- **Multi-query:** LLM генерирует N перефразировок запроса, результаты объединяются
- **Step-back prompting:** Из конкретного вопроса генерируется более общий, поиск ведётся по обоим

**Когда применять HyDE:**
- Запросы короткие и неинформативные
- Документы длинные и насыщенные деталями
- Dense retrieval показывает низкий recall

## 5. Reranking

### 5.1 Cross-Encoder Reranking

**Описание:** Пара (запрос, документ) передаётся в cross-encoder-модель, которая возвращает score релевантности. Более точно, чем bi-encoder, но медленнее.

**Модели:**
| Модель | Размер | Языки | Скорость | Качество |
|--------|--------|-------|----------|----------|
| cross-encoder/ms-marco-MiniLM-L-6-v2 | 66 MB | EN | Высокая | Хорошее |
| cross-encoder/ms-marco-MiniLM-L-12-v2 | 134 MB | EN | Средняя | Лучше |
| BAAI/bge-reranker-large | 560 MB | ZH+EN | Низкая | Отличное |
| cross-encoder/mmarco-mMiniLMv2-L12-H384 | 134 MB | Multilingual | Средняя | Хорошее |

**Параметры:**
- `top_k_rerank`: сколько кандидатов из retrieval передать на reranking (обычно top_k * 3)
- `top_k_final`: финальный top_k после reranking

**Ограничение:** Линейная сложность — reranking 20 кандидатов занимает ~100–500 мс на CPU.

---

### 5.2 LLM Reranking

**Описание:** LLM оценивает релевантность каждого документа или сортирует список документов по релевантности к запросу.

**Подходы:**
- **Pointwise:** LLM даёт score каждому документу отдельно (Yes/No или 1–5)
- **Listwise:** LLM переупорядочивает весь список (RankGPT)

**Когда применять:**
- Высокие требования к качеству, latency некритична
- Cross-encoder не поддерживает нужный язык
- Есть доступ к быстрому LLM (Ollama локально)

**Trade-off:** Высокое качество, но latency 1–5 сек на reranking.

---

### 5.3 ColBERT (поздняя интеграция)

**Описание:** Каждый токен запроса и документа кодируется отдельно, score — максимум косинусных совпадений (MaxSim).

**Преимущества:**
- Лучший баланс скорость/качество среди reranker-подходов
- Предвычисленные эмбеддинги документов — быстрее cross-encoder
- Поддерживается в Qdrant через multi-vector

**Модели:** `colbert-ir/colbertv2.0`, `BAAI/bge-m3` (в режиме ColBERT)

## 6. Метрики качества

### 6.1 Офлайн-метрики (Retrieval)

**recall@k** — доля релевантных документов, найденных в top-k результатах.
```
recall@k = |relevant ∩ retrieved_top_k| / |relevant|
```
Целевое значение: >= 0.8 для k=10. Низкий recall → проблема на уровне retrieval или chunking.

**precision@k** — доля релевантных документов среди top-k результатов.
```
precision@k = |relevant ∩ retrieved_top_k| / k
```
Целевое значение: >= 0.6 для k=5. Низкая precision → много нерелевантного шума в контексте.

**MRR (Mean Reciprocal Rank)** — средний обратный ранг первого релевантного документа.
```
MRR = (1/|Q|) * Σ 1/rank(first_relevant)
```
Целевое значение: >= 0.7. Низкий MRR → релевантный документ есть, но не на первом месте.

**NDCG@k (Normalized Discounted Cumulative Gain)** — учитывает порядок релевантных документов.
Целевое значение: >= 0.75 для k=10. Лучше MRR для многодокументных запросов.

---

### 6.2 Офлайн-метрики (Generation)

**Faithfulness (верность)** — насколько ответ опирается только на предоставленный контекст.
```
faithfulness = подтверждённые_утверждения / всего_утверждений_в_ответе
```
Целевое значение: >= 0.9. Низкая faithfulness → галлюцинации.

**Answer Relevancy (релевантность ответа)** — насколько ответ отвечает на вопрос.
Целевое значение: >= 0.8.

**Context Precision** — доля контекста, реально использованного при генерации ответа.
Целевое значение: >= 0.7.

**Context Recall** — насколько полно контекст покрывает необходимую для ответа информацию.
Целевое значение: >= 0.8.

**Hallucination Rate** — доля ответов, содержащих факты, не подтверждённые контекстом.
Целевое значение: <= 0.05 (5%).

---

### 6.3 Онлайн-метрики (Production)

**Latency p95** — 95-й перцентиль времени ответа end-to-end.
Целевые значения:
- Агрессивное SLA: <= 2 сек
- Стандартное SLA: <= 5 сек
- Аналитические задачи: <= 15 сек

**Latency по этапам (типичное распределение):**
- Embedding запроса: 50–200 мс
- Vector search: 10–100 мс
- Reranking (CPU): 100–500 мс
- LLM generation: 500–5000 мс (зависит от модели и длины)

**User Satisfaction Rate** — доля положительных оценок от пользователей (thumbs up/down).
Целевое значение: >= 0.8.

**Click-through Rate** — доля пользователей, кликнувших на источник документа.
Косвенная метрика доверия к источникам.

## 7. Evaluation Frameworks

### 7.1 RAGAS

**Ссылка:** `pip install ragas`

**Что измеряет:** Faithfulness, Answer Relevancy, Context Precision, Context Recall, Answer Correctness.

**Как работает:** Использует LLM-as-judge для оценки каждой метрики. Требует тестовый датасет с полями: question, answer, contexts, ground_truth.

**Особенности:**
- Не требует ручной разметки — LLM оценивает автоматически
- Поддерживает кастомные LLM-судьи (Ollama, OpenAI, Anthropic)
- Интеграция с LlamaIndex из коробки

**Пример использования:**
```python
from ragas import evaluate
from ragas.metrics import faithfulness, answer_relevancy

result = evaluate(dataset, metrics=[faithfulness, answer_relevancy])
```

---

### 7.2 TruLens

**Ссылка:** `pip install trulens-eval`

**Что измеряет:** Groundedness, Context Relevance, Answer Relevance (триада RAG).

**Особенности:**
- Dashboard для визуализации метрик
- Трассировка каждого шага пайплайна
- Интеграция с LlamaIndex и LangChain

---

### 7.3 DeepEval

**Ссылка:** `pip install deepeval`

**Что измеряет:** Широкий набор метрик: Faithfulness, Hallucination, Contextual Precision/Recall, Answer Relevancy, Custom metrics.

**Особенности:**
- Поддерживает CI/CD интеграцию (assert-based API)
- Кастомные метрики через GEval (LLM-as-judge с произвольным критерием)
- Поддержка локальных моделей через Ollama

---
