# Проект: Инструмент менеджмента с LLM + SQL

## Epic 1: Модели и структуры данных

**Story 1.1:** Реляционная модель для бизнес-требований
- Задача 1.1.1: Определить атрибуты бизнес-требования (ID, описание, приоритет, источник, статус, версионность)
- Задача 1.1.2: Спроектировать связи между бизнес-требованиями (иерархия, зависимости)
- Задача 1.1.3: Добавить поля для LLM-метаданных (эмбеддинг вектора, если потребуется для поиска)

**Story 1.2:** Реляционная модель для функциональных требований
- Задача 1.2.1: Связать функциональное требование с бизнес-требованием (внешний ключ)
- Задача 1.2.2: Добавить атрибуты: сценарий использования, предусловия, постусловия, acceptance criteria
- Задача 1.2.3: Предусмотреть поле для SQL-представления данных, к которым относится требование

**Story 1.3:** Реляционная модель для нефункциональных требований
- Задача 1.3.1: Определить типы NFR (производительность, безопасность, масштабируемость, доступность)
- Задача 1.3.2: Добавить атрибуты: метрика, целевое значение, текущее значение, метод измерения
- Задача 1.3.3: Связать NFR с компонентами системы (через foreign key или таблицу связей)

**Story 1.4:** Модель для трекинга задач
- Задача 1.4.1: Спроектировать таблицу `tasks` (статус, исполнитель, сроки, оценка, фактические часы)
- Задача 1.4.2: Связать задачи с любым типом требований (полиморфная связь или отдельные таблицы связей)
- Задача 1.4.3: Добавить историю изменений задачи (audit log)

**Story 1.5:** Общие структуры данных
- Задача 1.5.1: Определить, как будут храниться связи между требованиями разных типов
- Задача 1.5.2: Спроектировать индексы для частых запросов (по приоритету, статусу, исполнителю)
- Задача 1.5.3: Добавить поле `sql_snapshot` — предвычисленный SQL-запрос, который возвращает актуальные данные по требованию (экономия токенов)

## Epic 2: Конечный автомат (FSM) для стадий и пайплайнов

**Story 2.1:** Определение состояний (стадий работы)
- Задача 2.1.1: Выделить все стадии жизненного цикла проекта, для которых нужны разные пайплайны
- Задача 2.1.2: Описать возможные переходы между стадиями (какие переходы разрешены, какие запрещены)
- Задача 2.1.3: Определить начальное и конечное состояние

**Story 2.2:** Определение действий (пайплайнов)
- Задача 2.2.1: Для каждой стадии определить пайплайн — цепочку агентов или шагов, которая выполняется
- Задача 2.2.2: Задать входные и выходные данные для каждого пайплайна
- Задача 2.2.3: Определить условия перехода (например, пайплайн успешно завершён → смена стадии)

**Story 2.3:** Реализация FSM
- Задача 2.3.1: Выбрать способ реализации FSM (таблица в БД с текущим состоянием, библиотека, самописный enum-автомат)
- Задача 2.3.2: Реализовать механизм смены состояния через выполнение пайплайна
- Задача 2.3.3: Добавить валидацию: нельзя запустить пайплайн, не соответствующий текущей стадии

**Story 2.4:** Хранение состояний в SQL
- Задача 2.4.1: Создать таблицу `project_state` (project_id, current_stage, stage_history JSONB, metadata)
- Задача 2.4.2: Добавить триггеры для логирования переходов между стадиями
- Задача 2.4.3: Сделать SQL-функцию для получения допустимых пайплайнов для текущей стадии

## Epic 3: Детальная проработка пайплайнов

**Story 3.1:** Методология описания пайплайна
- Задача 3.1.1: Определить единый формат описания пайплайна (шаги, агенты, их роли, зависимости)
- Задача 3.1.2: Для каждого шага указать: входные данные, используемую LLM (или SQL), ожидаемый выход
- Задача 3.1.3: Описать fallback-сценарии (что делать при ошибке агента)

**Story 3.2:** Проработка пайплайна «Составление ТЗ»
- Задача 3.2.1: Расписать цепочку агентов: извлечение требований → классификация → приоритезация → сохранение в SQL
- Задача 3.2.2: Определить, какие данные передаются между агентами (формат, объём)
- Задача 3.2.3: Указать лимиты токенов для каждого агента в этом пайплайне

**Story 3.3:** Проработка пайплайна «Проектирование архитектуры»
- Задача 3.3.1: Расписать агентов: анализ NFR → генерация компонентов → валидация связей → SQL-схемы
- Задача 3.3.2: Определить точки, где результат может быть использован без LLM (через SQL-запросы)
- Задача 3.3.3: Добавить шаг с генерацией миграций БД на основе архитектуры

**Story 3.4:** Проработка пайплайна «Разработка фичи»
- Задача 3.4.1: Расписать агентов: декомпозиция фичи на задачи → генерация тестов → код-ревью через LLM
- Задача 3.4.2: Связать шаги пайплайна с функциональными требованиями из SQL
- Задача 3.4.3: Определить метрики успеха пайплайна (сколько задач закрыто, сколько токенов потрачено)

**Story 3.5:** Кросс-пайплайновые механизмы
- Задача 3.5.1: Реализовать кэширование результатов агентов на уровне SQL (повторное использование)
- Задача 3.5.2: Создать SQL-функции, которые могут вызываться внутри любого пайплайна (общие утилиты)
- Задача 3.5.3: Добавить логирование выполнения каждого шага пайплайна в отдельную таблицу

## Epic 4: Учёт расхода токенов и проверка гипотезы

**Story 4.1:** Базовый механизм учёта токенов
- Задача 4.1.1: Для каждого запроса к LLM считать входные и выходные токены (используя токенизатор конкретной модели)
- Задача 4.1.2: Сохранять токены в БД с привязкой к задаче, требованию, пайплайну, агенту
- Задача 4.1.3: Добавить поля: модель, timestamp, стоимость в $ (если есть цена за 1K токенов)

**Story 4.2:** SQL-экономия — подготовка к проверке гипотезы
- Задача 4.2.1: Замерить расход токенов в текущей версии (без оптимизаций SQL) на типовом сценарии
- Задача 4.2.2: Реализовать стратегии экономии через SQL: предвычисленные запросы, агрегации вместо сырых данных
- Задача 4.2.3: Внедрить механизм, где LLM сначала генерирует SQL, а затем получает компактный результат (а не весь контекст)

**Story 4.3:** Проверка гипотезы
- Задача 4.3.1: Запустить тот же сценарий с включёнными SQL-оптимизациями
- Задача 4.3.2: Сравнить расход токенов до и после (в абсолютных цифрах и процентах)
- Задача 4.3.3: Задокументировать результат: «Расход токенов снизился на X%»
- Задача 4.3.4: Если снижения нет или оно отрицательное — проанализировать причины (слишком сложные SQL, лишние вызовы)

**Story 4.4:** Дашборд и мониторинг
- Задача 4.4.1: Сделать SQL-запрос (или view) для отчёта по токенам за период
- Задача 4.4.2: Построить график расхода токенов по пайплайнам и агентам
- Задача 4.4.3: Настроить алерт при превышении дневного бюджета токенов

**Story 4.5:** Дополнительные оптимизации (после проверки гипотезы)
- Задача 4.5.1: Внедрить кэширование ответов LLM для повторяющихся запросов (на основе хеша промпта)
- Задача 4.5.2: Оптимизировать промпты — убрать лишние инструкции, использовать few-shot экономно
- Задача 4.5.3: Настроить автоматический выбор модели (дешёвая для простых SQL, дорогая для сложного анализа)
